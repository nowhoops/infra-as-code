# 1. Pro Git
- You can start using Git by: `git init`
- Cloning an existing repo: `git clone <url>`
- To add files (`git add`) and commit them (`git commit`)

---



- Files have two states:
  - Tracked (known to Git)
  - Untracked (not yet added)

### Workflow:
1. Modify files  
2. Stage changes (`git add`)  
3. Commit changes (`git commit`)  

### Key commands:

```
git status
```

→ shows file states

```
git diff
```

→ shows exact changes

### Other actions:
- Remove files: `git rm`
- Rename/move files: `git mv`

---



- `git log` shows commit history

### Useful options:

```
-p → show diffs
--oneline → compact view
--since / --until → date filtering
--author → filter by author
--grep → search commit message
-S → search content changes
```

---



### Amend last commit:
- `git commit --amend`

### Discard local changes:
- `git checkout -- <file>` or `git restore <file>`

!!! This deletes uncommitted work permanently !!!

---



Remotes are versions of your repo hosted elsewhere.

### View remotes:

```
git remote
git remote -v
```

### Add/remove remotes:

```
git remote add <name> <url>
git remote remove <name>
git remote rename <old> <new>
```

### Fetching vs Pulling:

- `git fetch <remote>`
→ downloads changes without merging

- `git pull`
→ fetches and merges automatically

### Pushing:

- `git push <remote> <branch>`
→ uploads your commits

### Deleting tags:

```
Local: git tag -d <tag>
Remote: git push origin --delete <tag>
```

---


- Work with local repositories  
- Track and commit changes  
- View history  
- Collaborate via remotes (push, pull, fetch)  
- Mark releases with tags  
- Customize workflow with aliases

# 2 Getting Started – What is Git?

### Core idea
- Git is a version control system that keeps track of changes in your project. Instead of storing differences between files like older systems, it saves snapshots of the whole project over time.

### Key concept
- Each commit is basically a snapshot of all your files at that moment.
- If something hasn’t changed, Git just points to the existing version instead of saving it again, so it stays efficient.

### Mostly local operations
- Git keeps the entire project history on your machine.
- That means you can view history, compare changes, and make commits without internet access.
- You only need to connect when you want to push or pull changes.

### Data is rarely lost
- Git is designed to mostly add data rather than delete it.
- Once something is committed, it’s usually very difficult to lose.

### States of files
1. Modified – you’ve changed the file, but Git hasn’t saved it yet  
2. Staged – the file is marked to be included in the next commit  
3. Committed – the file is saved in Git’s history
