## 2.1 Getting a Git Repository

- You can start using Git by: `git init`
- Cloning an existing repo: `git clone <url>`
- To add files (`git add`) and commit them (`git commit`)

---

## 2.2 Recording Changes

- Files have two states:
  - Tracked (known to Git)
  - Untracked (not yet added)

### Workflow:
1. Modify files  
2. Stage changes (`git add`)  
3. Commit changes (`git commit`)  

### Key commands:

```
git status
```

→ shows file states

```
git diff
```

→ shows exact changes

### Other actions:
- Remove files: `git rm`
- Rename/move files: `git mv`

---

## 2.3 Viewing History

- `git log` shows commit history

### Useful options:

```
-p → show diffs
--oneline → compact view
--since / --until → date filtering
--author → filter by author
--grep → search commit message
-S → search content changes
```

---

## 2.4 Undoing Changes

### Amend last commit:
- `git commit --amend`

### Discard local changes:
- `git checkout -- <file>` or `git restore <file>`

!!! This deletes uncommitted work permanently !!!

---

## 2.5 Working with Remotes

Remotes are versions of your repo hosted elsewhere.

### View remotes:

```
git remote
git remote -v
```

### Add/remove remotes:

```
git remote add <name> <url>
git remote remove <name>
git remote rename <old> <new>
```

### Fetching vs Pulling:

- `git fetch <remote>`
→ downloads changes without merging

- `git pull`
→ fetches and merges automatically

### Pushing:

- `git push <remote> <branch>`
→ uploads your commits

### Deleting tags:

```
Local: git tag -d <tag>
Remote: git push origin --delete <tag>
```

---

## 2.8 Summary

- Work with local repositories  
- Track and commit changes  
- View history  
- Collaborate via remotes (push, pull, fetch)  
- Mark releases with tags  
- Customize workflow with aliases  

---

## 1.3 Getting Started – What is Git?

### Core idea
- Git is a version control system that tracks changes. Unlike older systems, it stores data as snapshots of the entire project, not as a list of file changes.

### Key concept
- Git stores a snapshot of all files at each commit.
- If files haven’t changed, Git reuses the existing version (efficient storage).

### Mostly local operations
- Git stores full project history locally.
- Most actions (history, comparison, commits) work offline.
- You can push when internet is available, so no constant server connection is needed.

### Data is rarely lost
- Git mainly adds data rather than deleting it.
- Once committed, changes are very hard to lose.

### States of files
1. Modified – File has been changed but not yet saved in Git  
2. Staged – File is marked for the next commit  
3. Committed – File is safely stored
